⚠️ Why AutoADA Was Necessary

94.8% of the top million websites fail at least one WCAG criterion. The average site has 51 accessibility errors per page. Existing tools either produce overwhelming false positives, miss hidden violations behind popups and collapsed content, or simply fail on modern web architectures. AutoADA was built to address these real gaps with engineering precision.

⚖

Legal Compliance Urgency

ADA Title III lawsuits exceeded 4,000+ filings per year. Organizations need reliable, automated scanning to identify violations before legal action occurs — not after.

🎯

False Positive Problem

Existing scanners report color-contrast failures on elements with background images, flag decorative alt="" images, and generate noise that erodes trust. Confidence scoring was essential.

🔒

Modern Web Complexity

SPAs built with React, Vue, and Next.js render content dynamically. Cloudflare blocks headless browsers. Accordion content hides violations. A scanner that only sees static HTML misses the majority of issues.

⚡

Audit Efficiency

Manual WCAG audits take 40+ hours per site. AutoADA scans in minutes with concurrent page processing, producing detailed reports instantly — freeing auditors to focus on the 60% of criteria requiring human judgment.

🔮

Score Transparency

Scores without context are meaningless. AutoADA separates confirmed violations from likely issues and needs-review items, provides industry benchmarks, and clearly states that automated scanning covers only 30–40% of WCAG.

📝

Actionable Reports

A list of rule IDs helps nobody. AutoADA maps every violation to its WCAG success criterion with legal relevance, provides fix guidance with before/after code, and estimates remediation effort.

Sub-Phase 2.2: Scoring Accuracy Overhaul

2 Tasks • Core Algorithm

The original scoring weighted by rule count, not node count — meaning a site with 1 rule affecting 500 elements scored the same as 1 rule affecting 1 element. The entire scanner was rewritten. Page loads dropped from 4 to 2 per URL by scanning before and after overlay dismissal in a single load. Popup dismissal was made intelligent: buttons are now verified to be inside modal containers before clicking, preventing accidental navigation.

🔥 Challenges We Solved

Each of these problems was a real blocker discovered during development. They represent the difference between a prototype that works on example.com and a production scanner that handles the real web.

Challenge #1

Cloudflare Bot Detection

Sites behind Cloudflare showed "Just a moment..." challenge pages to Puppeteer. The scanner detected meta-refresh on the challenge page instead of real violations. Score was 18 (fake) instead of 28 (real).
Integrated puppeteer-extra with stealth plugin to evade headless detection. Added a challenge resolution detector that polls for up to 15 seconds for Cloudflare to clear, checking title, DOM elements, and meta-refresh patterns.
Impact: Sites like hereticparfum.com went from fake results to real 6-violation, 33-pass audits.
Challenge #2

Double-Penalization Scoring Bug

During the scoring rewrite, severity was being weighted in both the base score AND the penalty calculation. This produced scores of 0 for any site with even minor violations — completely unusable.
Isolated severity to the penalty calculation only. Base score now uses node count weighting while penalties apply logarithmic severity scaling. Verified with a 56-point gap test: same violations scored 56 at low confidence vs 0 at high.
Impact: Scoring now accurately reflects violation severity without mathematical artifacts.
Challenge #3

False Positive Epidemic

color-contrast violations on elements with background images, decorative alt="" images flagged as missing alt text, and ARIA hidden elements with tabindex=-1 reported as focus traps. Noise destroyed report credibility.
Built a multi-layer confidence system: rule-level classification (deterministic vs. heuristic), node-level context checks (inspecting actual CSS properties, DOM structure), and confidence-weighted scoring where low-confidence violations penalize only 15% vs 100% for confirmed.
Impact: Reports went from "noisy and unreliable" to "trustworthy with clear confidence levels."
Challenge #4

Gray/Empty Screenshots

Request interception blocked images for performance — but this meant screenshots showed gray boxes instead of actual page content. Additionally, marketing popups (Klaviyo, Shopify) appeared in every screenshot.
Removed images from the blocked resource types list. Added Klaviyo/Shopify-specific dismiss selectors. Rewrote overlay dismissal with a 4-strategy approach: overlay-checked click, relaxed click, accept fallback, and DOM force-removal of fixed overlays with z-index > 999.
Impact: Screenshots went from 0KB empty images to 330–390KB rich captures showing actual page content.
Challenge #5

Keyboard Trap False Positives

Keyboard trap detection used viewport-relative coordinates, so scrolled-off elements at the same screen position were falsely identified as the same element in a "trap." Additionally, pages that navigated during Tab cycling caused "Execution context destroyed" crashes.
Switched to absolute page positions using scrollY/scrollX. Added minimum 3-unique-elements check. Added Shift+Tab escape verification (real traps don't escape). Wrapped each Tab cycle in try/catch with context-destruction detection for graceful abort.
Impact: Zero false keyboard traps on GOV.UK, a11yproject, and example.com — all clean.
Challenge #6

Sitemap Blocked by Cloudflare

Sites behind Cloudflare or with restricted sitemaps returned HTTP 403 to Node.js default user-agent. Result: only 1 page scanned. The --crawl flag existed but was not discoverable.
Added realistic Chrome user-agent headers to sitemap fetching. Rewrote link crawling from Puppeteer-based to lightweight HTTP-based BFS (faster, more reliable). Auto-enabled crawling when sitemap returns 0 pages — no flag needed.
Impact: hereticparfum.com went from 0 discovered pages to 20 pages via sitemap.

💡 Key Design Decisions & Why

Every significant technical decision in AutoADA was made deliberately to solve a specific problem. Here is the reasoning behind the most important choices.

◆

Why Confidence-Weighted Scoring?

Because a color-contrast violation on an element with background-image is almost certainly a false positive. Without confidence weighting, a site with 50 contrast false positives and 0 real issues would score 20/100. With confidence weighting, low-confidence violations penalize only 15%, so the same site scores 85+. This matches reality.

◆

Why Puppeteer-Extra with Stealth?

A significant percentage of modern websites use Cloudflare or similar WAFs that detect and block headless browsers. Standard Puppeteer is immediately identified and served challenge pages. The stealth plugin patches Chrome's navigator properties, WebGL, and other fingerprinting vectors to appear as a regular browser. Without it, AutoADA would be unable to scan a large portion of the web.

◆

Why SPA Framework Detection?

React, Vue, and Next.js render content asynchronously. A scanner that runs immediately after DOMContentLoaded misses dynamically rendered content. AutoADA detects the framework in use and employs a MutationObserver-based DOM stability check — waiting until 1 full second passes with no DOM mutations. This ensures the scan captures the fully-rendered page state.

◆

Why Dual Viewport Testing?

Responsive designs frequently introduce accessibility regressions at mobile breakpoints: hamburger menus without keyboard support, touch targets smaller than 24px, and content hidden behind collapsed navigation. Scanning at both 1280px desktop and 375px mobile viewports catches issues that single-viewport scanners miss entirely.

◆

Why the Score Disclaimer?

Automated scanning can only evaluate 30–40% of WCAG 2.2 criteria. The remaining 60%+ require human judgment (screen reader testing, cognitive load, content clarity). AutoADA prominently disclaims this in every report and dashboard view. Transparency builds trust; overpromising destroys it.
